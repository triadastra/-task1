# 国债 ETF 3 秒快照数据解读指南

> 基于 `a_etf_cleaning_and_charts.ipynb` 的数据清洗与可视化流程整理。

---

## 1. 数据结构总览

- **来源**：上交所国债 ETF 3 秒级行情快照
- **标的**：`511090`、`511130`
- **核心字段**：价格（last/open/high/low/close）、盘口（5 档 bid/ask）、成交量（volume/amount/num_trades）、交易阶段（trading_phase_code）
- **时间粒度**：3 秒一条，盘中连续生成

---

## 2. 交易阶段：解读的入口

每个快照都有 `trading_phase_code`，不同阶段的字段有效性完全不同：

| 代码 | 含义 | 价格字段状态 | volume 含义 | 使用建议 |
|------|------|-------------|-------------|----------|
| **S10** | 开盘前（9:15 前） | 全为 0 → NaN | 0 | 无价格信息，仅作时间标记 |
| **C111** | 开盘集合竞价（9:15-9:25） | last/bid/ask 全为 0 → NaN | 集合竞价累计成交量 | 不能画价格线，末行 close 为开盘价 |
| **T111** | 连续竞价（9:30-11:30, 13:00-15:00） | 完整有效 | **日内累计成交量**（单调递增） | **主力分析区间**，所有技术指标基于此 |
| **E110** | 收盘集合竞价（15:00 后） | last / close 有效 | 收盘竞价累计 | **精确收盘价在此阶段最终确定** |

### ⚠️ 关键认知
- T111 最后一跳的 `last` **不等于** 当日收盘价，精确收盘价应从 **E110 末行的 `close`** 获取
- T111 阶段内的 `close` 字段在原始快照中始终为 0（交易所不提前回填），这是正常现象

## 3. 完整清洗流程

### 阶段一：单文件清洗（`read_one`）

对每一个原始 CSV 依次执行：

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | `pd.read_csv(path, low_memory=False)` | 读取原始 3 秒快照 |
| 2 | 删除 `Unnamed` 列、空名列、列名去空格 | 清除导出残留与格式问题 |
| 3 | 若存在 `iopv` 则整列删除 | 该字段原始全为 0，无信息价值 |
| 4 | 添加 `source_file`、`symbol_dir` | 便于溯源与调试 |
| 5 | `code`、`trading_phase_code` 去空格并转 `string` | 统一字符串格式，防止匹配失败 |
| 6 | `trade_time` 转 `datetime`（`errors='coerce'`） | 统一时间戳，错误值置为 NaT |
| 7 | `PRICE_COLS + VOLUME_COLS` 全部转 `numeric` | 确保数值型字段类型正确 |
| 8 | **价格字段 `0 → NaN`** | `last/open/high/low/close/bid/ask/high_limited/low_limited` 中的 0 视为无效 |
| 9 | 按 `code + trade_time` 去重，保留第一条 | 消除数据源重复导出 |
| 10 | **删除 `last` 为 NaN 的行** | 即剔除 S10 / C111 等价格全无效的阶段 |
| 11 | 提取 `date`、标记 `is_continuous`（`T111` 为 True） | 为后续阶段筛选与分组做准备 |

### 阶段二：多文件合并（`read_symbol`）

| 步骤 | 操作 | 目的 |
|------|------|------|
| 12 | 按标的目录（如 `511090`）筛选所有 CSV | 聚合单只 ETF 的全部日期 |
| 13 | `pd.concat` 合并，按 `code / trade_time / source_file` 排序 | 生成单只 ETF 的完整时序 |

### 阶段三：全局拼接

| 步骤 | 操作 | 目的 |
|------|------|------|
| 14 | `pd.concat([read_symbol('511090'), read_symbol('511130')])` | 合并两只 ETF 的清洗后数据，得到 `etf_all` |

### 阶段四：OHLCV 聚合（`make_ohlcv`）

| 步骤 | 操作 | 目的 |
|------|------|------|
| 15 | 按 `trading_phase_code` 筛选（默认 `T111`） | 价格分析只取连续竞价阶段 |
| 16 | 计算 `vol_inc`、`amt_inc` | 日内累计 volume/amount 做 `diff().clip(lower=0)`，得到每 3 秒真实成交 |
| 17 | 按 `code` 分组，`resample(freq)` 聚合 | 3 秒快照 → 1min / 5min / 1h / 1D 等 |
| 18 | 聚合规则：`open=first(last)`、`high=max(last)`、`low=min(last)`、`close=last(last)`、`volume=sum(vol_inc)` | 生成标准 OHLCV |
| 19 | 计算 `return`（`close.pct_change()`）与 `cum_return` | 得到收益率与累计收益率 |

### 阶段五：保存输出

| 步骤 | 操作 | 目的 |
|------|------|------|
| 20 | 按 `code` 分组，保留全部字段（含 NaN），输出到 `outputs/a_nan_marked/` | 保存轻量清洗结果，不填补、不删除阶段 |

### ⚠️ 清洗影响说明

- **S10 / C111 阶段被整体移除**：因为 `last` 为 NaN，这些行在步骤 10 被 `drop`。若需保留全量阶段做特殊分析，需注释掉该行。
- **`close` 在 T111 内始终为 NaN**：这是正常的（交易所日内不回填），做日 K 时应用 `last` 聚合，精确收盘价应从 E110 取。
- **`iopv` 已被整列删除**：如需保留，可去掉 `read_one` 中的删除逻辑。

## 4. 价格字段的 0 与 NaN

清洗策略：**价格字段的 0 视为无效，转为 NaN；成交量字段的 0 保留。**

| 字段 | 0 的含义 | 处理方式 |
|------|---------|----------|
| `last/open/high/low` | 当前阶段无效 | `0 → NaN` |
| `close`（T111 内） | 日内不更新，收盘后回填 | `0 → NaN`，但**不代表错误** |
| `bid_price1~5/ask_price1~5` | 无连续竞价盘口 | `0 → NaN` |
| `high_limited/low_limited` | 未设置涨跌停 | `0 → NaN` |
| `iopv` | 数据源未提供该字段 | 原始全 0 → NaN，可整列忽略 |
| `volume/amount/num_trades` | 真实无成交或初始状态 | **保留 0**，不转 NaN |

### 解读原则
看到价格字段为 0，先判断"这个字段在当前交易阶段是否应该有效"，再决定是数据缺失还是正常现象。

## 4. 成交量：累计值 vs 增量

这是最容易踩坑的地方。

- 原始快照中的 `volume/amount/num_trades` 是**日内累计值**
- 特征：T111 阶段单调递增，跨日归零，无成交时保持不变
- **不能直接作为 K 线成交量**，必须先计算增量

### 增量计算方法
```python
# 按 code + date 分组，计算相邻快照的差值
df['vol_inc'] = df.groupby(['code', 'date'])['volume'].diff().clip(lower=0).fillna(s)
df['amt_inc'] = df.groupby(['code', 'date'])['amount'].diff().clip(lower=0).fillna(s)
```

### 解读信号
| 现象 | 含义 |
|------|------|
| 相邻快照 volume 相同 | 这 3 秒内无成交 |
| volume 突然跳降 | 跨日了（新的一天从 0 开始累计） |
| volume 持续增长但幅度小 | 流动性清淡 |
| 尾盘 volume 激增 | 可能和 E110 收盘竞价有关 |

## 5. 盘口数据解读

数据包含 5 档盘口，可用于观察微观流动性：

| 现象 | 解读 |
|------|------|
| `bid_price1 > ask_price1` | **异常**，理论上不可能持续存在。若出现，检查是否为数据错误或集合竞价瞬时状态 |
| 买卖价差极小 | 流动性好，国债 ETF 通常如此 |
| 盘口深度突然放大 | 可能有机构大单挂单 |
| 盘口一侧突然清空 | 大单吃掉或撤掉，关注价格冲击 |

## 6. 数据质量检查清单

运行 notebook 中的 quality 单元格后，关注以下指标：

| 指标 | 正常预期 | 异常意味着什么 |
|------|---------|--------------|
| `last_nan_rate`（T111 阶段） | **0%** | 连续竞价期间价格缺失，数据异常 |
| `duplicate_code_time` | **0** | 同一时刻同一代码有多条记录，数据源重复导出 |
| `bid_gt_ask` | **极少或 0** | 买方最高价 > 卖方最低价，数据错误或特殊撮合状态 |
| `last_outside_limit` | **0** | 价格超出涨跌停限制，数据错误 |
| 每日快照数量 | 约 4800 条/日（4h × 3600s / 3s） | 明显偏少说明数据丢失 |

## 7. 图表解读逻辑

| 图表类型 | 解读重点 |
|---------|---------|
| **收盘价折线图** | 长期趋势方向、波动率大小、关键点位支撑/压力 |
| **K 线图** | 结合 O/H/L/C 看多空力量；长上影线 = 上方抛压大；长下影线 = 下方承接强 |
| **成交量柱状图** | 放量上涨 = 健康；缩量回调 = 健康；放量滞涨 = 警惕 |
| **折柱图（价格+成交量）** | 量价配合关系，识别背离（价涨量缩 / 价跌量增） |
| **RSI** | >70 超买（可能回调），<30 超卖（可能反弹）。国债 ETF 波动较小，RSI 通常不会极端 |
| **布林带** | 价格沿上轨运行 = 强势；跌破下轨 = 可能超跌反弹。国债 ETF 通常围绕中轨波动 |
| **收益率直方图** | 分布形态（是否对称、有无肥尾）、均值偏离程度 |
| **累计收益率对比** | 两只 ETF 的相对强弱，哪只跑得好、回撤控制如何 |

## 9. 使用建议

- **做价格分析（K 线、技术指标）**：只取 `T111` 阶段，用 `last` 作为价格基准
- **做日 K 收盘价**：优先取 `E110` 阶段末行的 `close`，而非 `T111` 末行的 `last`
- **做成交量分析**：务必使用 `vol_inc / amt_inc`，不要用原始 `volume / amount`
- **做数据质量报告**：以 `code` + `date` 分组，检查缺失率、重复率、异常率
